# Prompting Guide for Clojure

Этот файл фиксирует практические рекомендации по работе с LLM при разработке на Clojure.
Цель простая: получать код и разборы, которые уважают стиль языка, существующую кодовую базу и REPL-driven workflow.

## Какая модель подходит лучше

- `gpt-5.4`:
  основной выбор для повседневной разработки на Clojure, рефакторинга, чтения незнакомого кода, написания тестов и правок по месту.
- `gpt-5.4-pro`:
  подходит для сложных задач, где важны глубина рассуждений и аккуратность, например для макросов, конкурентного кода, архитектурных изменений и разбора нетривиальных ошибок.
- `gpt-5-mini`:
  разумный вариант для быстрых и дешевых рутинных задач, например небольших правок, генерации тестовых данных и простых преобразований.

Если нет явной причины оптимизировать цену или время ответа, начинайте с `gpt-5.4`.

## Базовые принципы хорошего промпта

При постановке задачи полезно явно просить модель:

- сохранять idiomatic Clojure и не переносить Java/OOP-паттерны без необходимости;
- предпочитать `data-first` подход, простые функции и явный data flow;
- учитывать REPL-driven разработку и не предлагать лишние абстракции;
- ориентироваться на существующую структуру проекта, зависимости и naming conventions;
- минимизировать объем изменений и не переписывать рабочий код без причины;
- писать код, который легко читать, тестировать и развивать.

Полезная формулировка для большинства задач:

```text
Сохраняй idiomatic Clojure. Предпочитай простые pure functions, data-first flow и минимальные изменения.
Не тащи OOP-паттерны, если они не требуются существующим кодом.
Учитывай REPL workflow, текущие зависимости и стиль проекта.
```

## Что стоит говорить явно

### Просьбы, которые обычно улучшают результат

- `Сначала изучи существующий код и продолжай в его стиле.`
- `Не придумывай новые abstraction layers без необходимости.`
- `Если есть несколько вариантов, выбери самый простой, который хорошо читается.`
- `Сохрани совместимость с текущим API и поведением.`
- `Если меняешь код, объясни инварианты и риски регрессии.`
- `Для функций предпочитай маленькие композиционные блоки вместо крупных универсальных оберток.`
- `Для ошибок и отладки опирайся на stacktrace, входные данные и фактический control flow.`

### Что полезно запрещать

- `Не переписывай код в Java-стиле.`
- `Не вводи протоколы, multimethods или макросы без явной необходимости.`
- `Не заменяй понятный код на более абстрактный только ради "красоты".`
- `Не меняй имена, shape данных и публичные функции без причины.`
- `Не выдумывай зависимости и API, которых нет в проекте.`

## Шаблоны промптов

Ниже шаблоны, которые можно копировать почти без изменений.

### 1. Новая функция

```text
Нужно добавить новую Clojure-функцию.

Контекст:
- Назначение: <что должна делать функция>
- Вход: <shape входных данных>
- Выход: <shape результата>
- Ограничения: <инварианты, ошибки, производительность>

Требования:
- Сохраняй idiomatic Clojure и data-first style.
- Предпочитай простую pure function без лишних abstraction layers.
- Не вводи новые зависимости.
- Если уместно, добавь короткие тесты.

Сначала кратко опиши подход, затем предложи код.
```

### 2. Рефакторинг

```text
Нужно отрефакторить существующий Clojure-код без изменения поведения.

Требования:
- Сохрани текущее поведение и публичный API.
- Уменьши сложность и дублирование, но без overengineering.
- Сохраняй idiomatic Clojure.
- Не вводи OOP-style abstractions.
- Отдельно перечисли риски регрессии и что нужно проверить тестами.

Сначала объясни, что именно стоит упростить, затем предложи минимальный патч.
```

### 3. Макросы

```text
Нужно помочь с Clojure-макросом.

Требования:
- Сначала объясни, нужен ли здесь макрос вообще.
- Если макрос нужен, разложи отдельно expansion, runtime behavior и ограничения.
- Избегай сложной магии, если задачу можно решить обычной функцией.
- Покажи пример macroexpand и объясни риски для отладки и сопровождения.

Ответ строй так, чтобы им можно было пользоваться при REPL-driven разработке.
```

### 4. Code review

```text
Сделай review Clojure-кода.

Фокус:
- bugs и поведенческие регрессии;
- нарушения idiomatic Clojure;
- ненужные абстракции;
- проблемы с shape данных, nil-handling, sequence semantics и lazy behavior;
- пробелы в тестах.

Сначала выдай findings по убыванию серьезности, затем коротко дай summary.
```

### 5. Отладка ошибки или падающего теста

```text
Помоги разобрать ошибку в Clojure-коде.

Данные:
- stacktrace: <вставить>
- failing test / вход: <вставить>
- ожидаемое поведение: <описать>
- фактическое поведение: <описать>

Требования:
- Не гадай без опоры на stacktrace и код.
- Сначала перечисли наиболее вероятные причины.
- Затем предложи минимальный способ локализовать проблему.
- После этого предложи исправление с минимальной областью изменений.
```

### 6. Работа с незнакомым кодом

```text
Нужно разобраться в незнакомом Clojure-коде и предложить изменение.

Требования:
- Сначала опиши текущий data flow и ключевые точки входа.
- Не делай архитектурных выводов без опоры на реальные namespace и вызовы.
- Если чего-то не хватает, явно перечисли неизвестные.
- Предлагай изменение только после краткого анализа текущего поведения.

Сохраняй стиль существующего проекта и избегай лишних перемещений кода.
```

## Чеклист перед отправкой промпта

Перед тем как отправить запрос модели, проверь:

- указана ли конкретная цель изменения;
- описан ли shape входных и выходных данных;
- зафиксированы ли инварианты и ограничения;
- сказано ли, что нужно сохранить idiomatic Clojure;
- запрещены ли ненужные абстракции и OOP-style решения;
- понятен ли допустимый объем изменений;
- обозначено ли, нужны ли тесты, объяснение или только патч.

## Короткая рекомендуемая заготовка

Если не хочется писать длинный промпт, можно использовать такой базовый шаблон:

```text
Помоги с задачей в Clojure.
Сначала изучи существующий код и держись его стиля.
Сохраняй idiomatic Clojure, data-first flow и минимальную область изменений.
Не тащи OOP-паттерны и не вводи лишние abstraction layers.
Если есть trade-offs, выбери самый простой и читаемый вариант.
Если меняешь поведение, явно это отметь.
```
