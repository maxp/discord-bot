# Project Rules

Этот документ фиксирует базовые технические правила проекта.
Цель этих правил: уменьшить случайные архитектурные расхождения и сделать проект предсказуемым для разработки и сопровождения.

## Language and Runtime

- Проект использует последнюю стабильную версию Clojure.
- Обновление версии Clojure должно выполняться отдельным понятным изменением, а не прятаться внутри других задач.
- После обновления версии Clojure нужно проверять базовый запуск проекта, REPL и минимальный smoke path бота.
- Если для конкретной зависимости или окружения требуется временно отстать от latest stable, это должно быть явно задокументировано.

## Dependency Management

- Управление зависимостями выполняется через `deps.edn`.
- Новые runtime, dev и test зависимости должны добавляться в `deps.edn`.
- Алиасы для разработки, тестов, линтинга, сборки и REPL должны описываться в `deps.edn`.
- Не следует вводить альтернативный основной инструмент управления зависимостями вроде Leiningen, если проект явно не принял такое решение отдельным ADR.
- Логика зависимостей и алиасов не должна дублироваться в shell-скриптах или документации как единственный источник правды.

## Developer Commands

- `Makefile` является основным интерфейсом для типовых команд разработчика.
- Повторяемые команды должны запускаться через `make`, а не только через длинные одноразовые shell-команды в документации.
- Минимальный ожидаемый набор целей `Makefile` после появления runtime-кода:
  - `make dev`
  - `make run`
  - `make test`
  - `make lint`
- Внутри `Makefile` допускается вызывать `clojure -M`, `clojure -X` и project aliases из `deps.edn`.
- Если появляется новая обязательная команда для ежедневной разработки, для нее нужно добавить цель в `Makefile`.

## Project Structure

- Документация проекта хранится в `docs/`, а `README.md` остается входной точкой и индексом.
- Архитектурные решения, которые важно объяснять через "почему", должны фиксироваться в `docs/decisions/`.
- Временные локальные артефакты и служебные файлы инструментов не должны попадать в репозиторий без явной причины.
- Локальное окружение проекта хранится в файле `.env`.
- Файл `.env` должен существовать всегда в локальной рабочей копии проекта.
- Секреты и локальные значения из `.env` не должны коммититься в репозиторий.

## Code Style

- Между top-level функциями должно быть две пустые строки.
- Правила код-стайла нужно соблюдать в новом коде и поддерживать при редактировании существующего кода.
- Для доступа к вложенным полям структуры по ключам используй `(some->)` вместо `get-in`.
  - `(some-> m :data :id)` вместо `(get-in m [:data :id])` — это короче, лениво и безопасно при nil на любом уровне.
- При проверке на отсутствие обязательных значений нельзя просто гасить исключение при помощи `(when)` или `(when-let)` — обязательно исключение с контекстом.
  - `(when-let [x (get m :key)] ...)` без `else` — недопустимо для обязательных полей.
  - Используй `(let [x (or (get m :key) (throw (ex-info "Missing required key" {:key :key})))] ...)` или аналогичную явную проверку.

## Documentation Discipline

- Если меняется способ запуска, тестирования, сборки или настройки окружения, соответствующая документация должна обновляться в том же изменении.
- Если правило проекта перестает быть верным, документ нужно менять сразу, а не "потом".
- Документация должна описывать норму проекта, а не случайное текущее состояние рабочей машины разработчика.

## Exceptions

- Любое отступление от этих правил допустимо только если:
  - есть явная техническая причина;
  - это решение зафиксировано в документации или ADR;
  - команда понимает, почему исключение лучше базового правила.
